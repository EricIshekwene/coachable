# Coachable v2

v2 is a complete rebuild — new repo, new frontend, new backend. Both `src/` and `server/` are written from scratch against the standards documented here. The only thing ported from v1 is the Slate canvas editor (`src/slate/`), which carries forward unchanged because it already works well.

The approach: define standards first, then execute against them. Design standards (spacing, typography, color, components) are written as documents before any component is built. Engineering standards (file structure, testing patterns, API contracts, security) are planned and documented before code is written. The goal is that by the time execution starts, the decisions are already made and the work is mechanical.

**What carries over from v1:**
- The Slate canvas editor (`src/slate/`) — ported as-is
- These planning and design docs (`v2/` folder)
- `server/db/schema.sql` — as a reference for the v2 schema (not run directly; v2 schema is rebuilt per `engineering/audits/api-review.md` recommendations)

**What is built fresh:**
- All of `server/` — rebuilt per the API review audit, with `/api/` prefix, split admin routes, proper migration system, background jobs out of the process, etc.
- All of `src/` except `src/slate/` — new component library, new pages, new routing, no MUI, lazy imports from day one

**v1 repo** stays live and untouched as prod until v2 is ready to cut over. No changes are made to v1 during v2 development.

This folder holds all of that: design standards, engineering standards, audits of the v1 codebase, and planning docs for each initiative. `TODO.md` is the ordered work list.

## Docs Convention (standing rule)

All documentation lives in `docs/`. This is enforced from the first commit — not migrated later.

**Banned directories:** No `.md` files in `src/` or `server/`. Markdown next to code is the v1 anti-pattern this structure was built to fix.

**Exempt:** `CLAUDE.md` at the repo root. It is an AI navigation index that must live at the repo root to be auto-loaded — it is not a feature doc.

**No other exemptions.** This includes `src/slate/` — any Slate internal docs (architecture notes, editor internals) go to `docs/engineering/slate/`, not inside the closed boundary.

**What counts as a doc:**
- Feature writeups, architecture notes, design decisions → `docs/`
- Investigation and root-cause writeups worth referencing again → `docs/engineering/audits/`
- One-off fix notes with no lasting reference value → git commit message only
- Bug reports → Notion (not a code docs concern)

**`docs/INDEX.md` maintenance:** The developer adding the doc adds the index entry in the same commit. A doc without an index entry is incomplete — same rule as a route without a test.

**CI enforcement:** A GitHub Actions step fails the build if any `.md` file is found inside `src/` or `server/`:
```bash
find src/ server/ -name "*.md" | grep -q . && echo "❌ .md files found in src/ or server/ — move to docs/" && exit 1 || exit 0
```
