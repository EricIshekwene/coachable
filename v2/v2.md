# Coachable v2

v2 is the initiative to mature Coachable from a working prototype into a production-grade product. The core features (play editor, playbooks, teams, notifications, sharing) are built and live. v2 is not a rewrite — it is the layer of engineering rigor, design consistency, and platform completeness that makes the existing product stable, scalable, and shippable to real users at scale.

## Design standards summary

The design standards in this folder are the visual law for every surface. Key decisions:

- **4px grid** — all spacing is a multiple of 4px, no exceptions
- **Six typography levels** — heading, subheading, label, body, secondary, caption; no ad-hoc sizes
- **Dark-first** — the app is dark mode by default; light mode support is a secondary concern
- **Manrope** — the type family for all UI text
- **BrandX tokens** — color system built on CSS custom properties; semantic tokens (`--color-error`, `--color-warning`, etc.) pending the color system initiative (1.1)

Surface-specific rules live in `design/desktop/` and `design/mobile/`. The general doc in `design/general-formatting-standards.md` applies everywhere and takes precedence when rules conflict.

## Engineering standards summary

- **File structure** — the v2 target layout is in `engineering/planning/architecture/proposed-file-structure.md`. The current structure has not been migrated yet (initiative 1.2).
- **Testing** — role-based UI tests co-locate with pages in `tests/` subfolders; server tests live in `server/tests/` mirroring the route structure. Both standards are fully written.
- **Backend** — route files are `.js` only; JSDoc on every handler; no markdown in `server/`. Conventions in `engineering/backend-code-standards.md`.
- **Frontend** — React/Vite; no MUI in v2; all shared components go in `src/ui/`; feature flags via `useFeatureFlags()` or `<FlagGate>`.

## How to use this folder

- **Starting a new initiative?** Check `TODO.md` for the group it belongs to and whether it has dependencies above it.
- **Making a UI change?** Read `design/general-formatting-standards.md` first, then the surface-specific doc.
- **Writing a test?** Read `engineering/planning/testing/ui-testing-standards.md` (frontend) or `server-testing-standards.md` (backend).
- **Adding a server route?** Read `engineering/backend-code-standards.md` and follow the JSDoc pattern in `engineering/planning/infrastructure/security-and-code-quality.md`.
- **Confused about where a file should go?** Read `engineering/planning/architecture/proposed-file-structure.md`.
