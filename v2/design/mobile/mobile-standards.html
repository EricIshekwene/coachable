01 — Spacing

Everything snaps to the 4px grid

All padding, gap, and margin values must be multiples of 4px. No exceptions.

A 4px grid creates consistent visual rhythm across all screens without over-constraining layout. 4px is the atom — structural gaps between components tend to use 8px+ multiples. Inner component padding can use 4px or 12px. "14px feels right" is not a valid reason — use 12px or 16px.

02 — Typography

Six named levels — no ad-hoc sizes

Every text element on every screen maps to one of six defined levels. Inventing intermediate sizes is not allowed.

A fixed scale creates consistency and forces priority decisions. If you're unsure between two sizes, ask whether this text is a heading or a label — then use that level. Manrope is for headings (it carries the brand). DM Sans is for everything else (it reads cleanly at small sizes). Minimum body text is 15px — below that, achieving WCAG AA contrast at small sizes becomes harder and text strains on mobile screens.

03 — Touch Targets

Every tappable element is at least 48×48px

The tap zone — not the visual size — must be 48×48px minimum. A small-looking icon button can still meet this with padding.

This satisfies WCAG 2.2 AA (2.5.8), Apple HIG (44pt), and Material Design 3 (48dp) simultaneously. A 24px icon with p-1 (4px padding) gives a 32px tap zone — half the minimum. A 24px icon with p-3 (12px padding) gives a 48px zone. The visual and the interactive layer are separate concerns.

04 — Color & Contrast

Text must meet 4.5:1 contrast. BrandGray2 is a border color, not a text color.

Normal text: 4.5:1 minimum against its background. Large text (18px+ or 14px+ bold): 3:1 minimum. (WCAG 2.2 AA)

BrandGray2 (#4b5157) is only 2.1:1 against BrandBlack — it fails for text. It's useful as a subtle border color but should never carry readable text. For muted text, use BrandGray (#9AA0A6) which passes at 5.8:1. BrandOrange on BrandBlack passes at 5.7:1, making it safe for active labels and links.

05 — Borders

1px only. A heavier border is not more defined — it's just louder.

All borders are 1px. No exceptions for cards, inputs, modals, or dividers.

On retina displays, 1px is already crisp and clear. 2px borders add visual weight that competes with content. The distinction between surfaces should come from background color contrast and shadow — not from border weight. A 2px border on a card signals "I couldn't make the surfaces look distinct, so I drew a thicker line." Fix the root cause instead.

06 — Border Radius

Radius signals hierarchy — use it consistently

Use the defined radius scale. Do not pick a radius because it "looks nice" — pick the one that matches the component's role.

Larger radius = more prominent, more floating. Smaller radius = tighter, more inline. A sheet that slides up from the bottom should feel like a separate layer — 20px top radius. A button is an inline control — 8px. A badge is a tiny inline label — 6px. Consistency across the codebase means users build spatial intuition about what type of surface they're looking at.

Example — radius scale by role

6px Chips, badges, small tags
8px Buttons, inputs, dropdowns
12px Cards, panels, list containers
16px Large overlays, floating elements
20px top Bottom sheets (top corners only)
pill Avatars, toggles, drag handles
07 — Elevation & Shadow

Shadow means "floating." Cards in a list are not floating.

Dark mode: cards use no shadow — background contrast does the work. Overlays (modals, dropdowns, toasts) always use a strong shadow. Light mode cards may use a soft shadow to compensate for near-identical backgrounds.

In dark mode, #2a2e34 (card) on #121212 (page) is clearly distinct — a shadow adds nothing and wastes GPU. On lower-end Android devices, compositing shadows on 20+ list items during scroll causes jank. Reserve shadows for the semantic purpose: "this element is physically above the page." A card in a list is part of the page. A modal is above it. That distinction matters.

08 — Vertical Rhythm

Gap size communicates grouping. More gap = less related.

8px between items in a group. 16px between sibling cards. 24px before a new named section.

Gestalt proximity: elements that are close together are perceived as related. The three-tier gap system (8 → 16 → 24) makes grouping legible without needing explicit divider lines. If everything has the same gap, nothing groups — the page reads as an undifferentiated list of items with no hierarchy.

09 — Layout & Margins

Mobile is a single column. 16px margins. Nothing goes edge-to-edge.

16px horizontal margin on every mobile screen. All content — including lists — sits inside that margin. No two-column layouts below 768px.

At 375px (iPhone SE: 320px) with 16px margins, the usable column is 343px (288px). Everything must fit in that column without horizontal scroll — WCAG 1.4.10 requires content to work at 320px viewport width. Side-by-side columns on mobile force content into columns too narrow to read comfortably. Lists that bleed edge-to-edge look uncontained and create an inconsistent layout rule that's hard to enforce.

10 — Active States

Active = color change. Not backgrounds, pills, or animation.

Active/selected states are communicated by color alone — BrandOrange on the active item, BrandGray on inactive. No filled background pill, no underline indicator, no animated bubble.

Color is the clearest, cheapest signal for "this is where you are." Pill backgrounds and animated indicators add complexity, require more vertical space, and can look out of place against the dark surface. Coachable's nav has five items — a pill behind an icon creates visual noise in a tight space. The most recognized mobile apps (Instagram, Twitter, YouTube) use icon+label color change with no additional decoration.

11 — Overlays & Sheets

Sheets grow with content, cap at 85dvh, and always signal dismissibility

Bottom sheets: content-driven height, max 85dvh, scrollable inside. Always show a drag handle. Close on backdrop tap or downward drag.

A fixed-height sheet wastes space for short content and clips long content. Content-driven height with a max cap handles both cases. The drag handle pill (36×4px) is a native iOS/Android convention — users expect it and know it means "drag to dismiss." Without it, dismissal feels ambiguous. The 85dvh cap ensures the top of the page remains partially visible, orienting the user to where they were.

Example — sheet structure and the drag handle

page content (dimmed)

Sheet title

Content drives height. If content is long, sheet grows to 85dvh and the content scrolls inside — not the page.

Drag handle — the small pill at the top. Signals "this panel is drag-dismissible." Not a close button — just a visual hint. Always present on sheets.

85dvh cap — the sheet never goes full-screen unless the content demands it. Keeping the page partially visible orients the user.

Scrolls inside — when content exceeds the cap, it scrolls within the sheet. The page behind stays fixed. Never two scroll containers fighting.

Desktop — the same sheet becomes a centered dialog (full border radius, max-w-md). One codebase, two presentations.
